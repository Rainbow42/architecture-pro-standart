| Код | Требования | Комментарий |
|-----|------------|-------------|
| F   | Функциональные (Functionality) | |
| F1 | Клиент видит на сайте список депозитов с актуальными ставками | Нужен единый источник ставок вместо Excel |
| F2 | Новый клиент на сайте подаёт заявку на депозит (телефон, ФИО); оператор кол-центра звонит и может предложить особые условия | Интеграция Сайт → CRM кол-центра |
| F3 | Клиент в интернет-банке видит депозиты с актуальными и персонализированными ставками | Read-only API ставок, персонализация по данным клиента |
| F4 | Клиент в интернет-банке подаёт заявку на депозит (счёт, сумма) с подтверждением СМС-кодом | СМС без доработки ядра подрядчика — отдельный модуль в ИБ |
| F5 | Менеджер бэк-офиса обрабатывает онлайн-заявку и подтверждает условия в АБС (MVP) | Заявки передаются в АБС асинхронно |
| F6 | Клиент получает СМС после подтверждения ставки и после открытия депозита | Интеграция с существующим СМС-шлюзом |
| F7 | Сотрудники бэк-офиса депозитов и кредитов работают со ставками в одной системе | Модуль ставок в АБС вместо XLS |
| F8 | На сайте новый клиент подаёт заявку на кредит; при заполнении паспортных данных — предварительное решение | Online-вызов скоринга + БКИ |
| F9 | Заявка с сайта продолжается в отделении по тому же ID | Сквозной идентификатор заявки |
| F10 | В интернет-банке клиент видит предодобренные кредитные предложения и подаёт заявку с СМС-подтверждением | Предодобрение — batch вне рабочих часов скоринга |
| F11 | Клиент получает СМС об изменении статуса кредитной заявки | Через СМС-шлюз |
| F12 | После заявки на кредит с сайта клиент получает СМС о необходимости прийти в отделение для идентификации и получения кредита | Аналогично депозитному сценарию для новых клиентов |
| F13 | Условия кредитных продуктов на сайте и в интернет-банке отображаются из Кредитного конвейера | Единый источник продуктовой информации |
| F14 | При подаче заявки по предодобренному предложению выполняется повторный скоринг (упрощённая процедура), если ситуация клиента изменилась | Online-вызов скоринга при подтверждении заявки |
| F15 | Оператор кол-центра видит актуальные ставки по депозитам | Read-only API; см. IT-landscape, перспектива после MVP |
| F16 | Партнёрский кол-центр получает актуальные ставки в виде файлов | SFTP-выгрузка; внешняя система без API |
| U   | Удобство использования (Usability) | |
| U1 | Интерфейс должен быть удобным для клиентов старшего возраста | Крупные элементы, понятные шаги, единый дизайн банка |
| U2 | Отклик операций — миллисекунды | Кэширование справочников и ставок в промежуточном сервисе |
| U3 | Единая система дизайна и брендирование во всех каналах | Согласование UI между сайтом и ИБ |
| R   | Надёжность (Reliability) | |
| R1 | Доступность сервисов 24/7, 99,9% | Горизонтальное масштабирование ИБ и нового сервиса заявок |
| R2 | При сбое ЦОД — переключение на резервный ЦОД | Для интернет-банка и сервиса заявок |
| R3 | АБС не должна падать из-за онлайн-нагрузки | Асинхронная интеграция, без прямого online-доступа ИБ к Oracle |
| R4 | Кредитный конвейер и система скоринга работают 24/7 и поддерживают горизонтальное масштабирование | Критичные системы кредитного контура |
| P   | Производительность (Performance) | |
| P1 | Справочные данные загружаются быстрее 1 секунды | Кэш в сервисе заявок / ИБ |
| P2 | Скоринг не нагружается предрасчётами в рабочие часы | Batch предодобрений — ночью |
| P3 | АБС масштабируется только вертикально | Ограничивает частоту синхронизации с АБС |
| P4 | Заявки по предодобренным кредитам обрабатываются в тот же рабочий день | Ожидание бизнеса; конфликтует с +R3 — нужен online-канал в конвейер |
| S   | Поддерживаемость (Supportability) | |
| S1 | Использовать знакомые технологии: .NET, Java, Oracle, MS SQL, PHP | Минимум новых стеков в MVP |
| S2 | Документация API и интеграций для дальнейшего расширения | ADR, OpenAPI для сервиса заявок |
| S3 | Изолированный модуль депозитов в интернет-банке (первый шаг к микросервисам) | Только для процесса депозитов, не весь монолит |
| S4 | Kafka — на перспективу, не в MVP | Текущая версия ИБ несовместима с Kafka |
| S5 | Новые технологии должны быть совместимы с существующими платформами разработки банка | Снижение рисков интеграции |
| S6 | При выборе стека учитывать наличие экспертизы внутри банка | Минимум обучения команд |
| +R  | + Ограничения (Restrictions) | |
| +R1 | Новый клиент проходит идентификацию только в отделении | Регуляторное требование банков |
| +R2 | Нельзя давать сотрудникам депозитов и кредитов прямой доступ к данным друг друга | Разделение в АБС, согласование через workflow |
| +R3 | Нельзя ускорить batch-обмен АБС ↔ Кредитный конвейер чаще 1 раз/сутки | Техническое ограничение текущей интеграции |
| +R4 | Партнёрский кол-центр не поддерживает API, только файлы по SFTP | Внешнее ограничение |
| +R5 | Доработка СМС в ядре подрядчика ИБ нежелательна | Реализация силами команды банка |
| +R6 | Прямая работа интернет-банка с API АБС в новом процессе нежелательна | Перегрузка Oracle |
| +R7 | Передача персональных данных с сайта и из интернет-банка — только по защищённым каналам (HTTPS/TLS) | Регуляторные и операционные требования безопасности |
