MVP онлайн-заявок на открытие депозита в интернет-банке и на сайте

### **Автор:**
Команда цифровой трансформации

### **Функциональные требования**

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| UC-1 | Клиент, Сайт, Сервис заявок на депозиты (DAS) | Просмотр депозитов на сайте | 1. Клиент открывает сайт.<br>2. Сайт запрашивает список депозитов и ставок у DAS.<br>3. Клиент видит актуальные ставки. |

| UC-6 | АБС, DAS, СМС-шлюз, Клиент | Уведомление клиента | 1. АБС передаёт результат в DAS.<br>2. DAS/АБС отправляет СМС о подтверждении ставки.<br>3. После открытия депозита в АБС — СМС об открытии. |
| UC-7 | Менеджер бэк-офиса, АБС | Ведение ставок | 1. Менеджер депозитов/кредитов вводит ставки в модуле АБС.<br>2. Ставки синхронизируются в DAS для каналов. |

|**№**|**Требование**|

| NFR-1 | Не подключать интернет-банк напрямую к БД АБС для нового процесса |

### **Решение**

#### Архитектурное решение

Вводится промежуточный **Deposit Application Service (DAS)** — сервис заявок на депозиты на Java Spring Boot + PostgreSQL (или MS SQL). Он:
- принимает заявки от сайта и интернет-банка;
- кэширует и отдаёт ставки из АБС;
- передаёт заявки в АБС асинхронно;
- не создаёт прямую нагрузку на Oracle от каждого клика клиента.

**Интернет-банк** получает новый модуль «Депозиты»: UI + модуль СМС-подтверждения + REST-клиент к DAS.

**АБС** получает модуль «Ставки депозитов» и экран обработки онлайн-заявок.

**Сайт** получает страницы депозитов (данные из DAS) и форму заявки (лиды в кол-центр).

#### Диаграммы

- Диаграмма контекста (C4): `c4_context_deposit.drawio`
- Диаграмма контейнеров (C4): `c4_container_deposit.drawio`

#### Основные компоненты и интеграции

| Компонент | Технология | Интеграции |
|-----------|------------|------------|
| Deposit Application Service | Java Spring Boot, PostgreSQL | ← ИБ, Сайт (REST); ↔ АБС (batch/REST); → СМС-шлюз |
| Интернет-банк (модуль депозитов) | ASP.NET MVC | → DAS (REST), → СМС-шлюз |
| АБС (модуль ставок и заявок) | Delphi, Oracle PL/SQL | ↔ DAS; → СМС-шлюз |
| Сайт | PHP, React | → DAS (read), → Кол-центр (заявки) |
| СМС-шлюз | Существующий | ← ИБ, DAS, АБС |

#### Логика принятия решений

1. **DAS вместо прямого доступа к АБС** — команда АБС указала на перегрузку Oracle; асинхронная интеграция снижает риск.
2. **Java для DAS** — в банке есть экспертиза (кол-центр, АБС, Camunda).
3. **Модуль в монолите ИБ, не полный микросервис** — MVP только для депозитов, без переписывания всего ИБ.
4. **Ставки в АБС** — там же согласование; Excel заменяется единым источником.
5. **СМС своими силами** — избегаем зависимости от подрядчика ядра ИБ.

### **Альтернативы**

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| Прямой доступ ИБ → БД АБС | Уже используется для платежей, но перегружает Oracle при масштабировании |
| Все заявки только через АБС API online | Команда АБС против online-нагрузки |
| Kafka для интеграции | ИБ несовместим с текущей версией Kafka |
| Полный перевод ИБ на микросервисы | Слишком дорого для MVP |

### **Недостатки, ограничения, риски**

- **Недостаток:** дополнительный сервис (DAS) — новая точка отказа. Нужен мониторинг и резервирование.
- **Ограничение:** бэк-офис всё ещё участвует в MVP — полная автоматизация только через год.
- **Ограничение:** новый клиент с сайта не может открыть депозит без визита в отделение.
- **Риск:** задержка синхронизации DAS ↔ АБС — клиент ждёт обработки дольше, чем при мгновенном online.
- **Риск:** команда ИБ и подрядчик — узкое место по UI и СМС-модулю.
