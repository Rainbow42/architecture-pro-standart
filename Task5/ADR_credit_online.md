### **Решение**

#### Архитектурное решение

Вводится **Credit Application Service (CAS)** — сервис заявок на кредиты на Java Spring Boot + PostgreSQL (или MS SQL), по аналогии с DAS:
- принимает заявки с сайта и интернет-банка;
- отдаёт условия кредитных продуктов из Кредитного конвейера;
- вызывает **скоринг online** только в момент подачи заявки;
- передаёт заявки **напрямую в Кредитный конвейер** (REST API Camunda);
- хранит сквозной ID заявки;
- инициирует СМС через шлюз.

**Предодобренные предложения:** ночной batch-job читает данные клиентов из АБС (batch, как сейчас), вызывает скоринг в off-peak, сохраняет предложения в CAS. Днём клиент видит их в ИБ без нагрузки на скоринг.

**Отделение:** после идентификации CAS синхронизирует заявку с АБС; менеджер продолжает работу по сквозному ID.

**Интернет-банк** получает модуль «Кредиты»: UI + СМС-модуль (силами банка) + REST-клиент к CAS.

#### Компоненты и интеграции

| Компонент | Технология | Интеграции |
|-----------|------------|------------|
| Credit Application Service | Java Spring Boot, PostgreSQL | ← Сайт, ИБ (REST); → Кредитный конвейер (REST); → Скоринг (REST); ↔ АБС (batch sync); → СМС-шлюз |
| Application API (CAS) | REST | Приём заявок, выдача продуктов и предодобрений |
| Pre-approval Batch (CAS) | Scheduled job | ← АБС (batch); → Скоринг (off-peak) |
| Кредитный конвейер | Camunda, Java | ← CAS (REST online); → Скоринг; ↔ АБС (batch 1/сут, +R3) |
| Скоринг | Python Flask | ← CAS, Конвейер; → БКИ (online); ← АБС (batch данные) |
| Интернет-банк (модуль кредитов) | ASP.NET MVC | → CAS (REST), → СМС-шлюз |
| Сайт | PHP, React | → CAS (REST) |
| СМС-шлюз | Существующий | ← CAS, ИБ, Конвейер |

#### Логика принятия решений

1. **CAS вместо прямого доступа ИБ/сайта к АБС и конвейеру** — снижает нагрузку на Oracle (+R6) и даёт единую точку для сквозного ID (F9).
2. **Online REST CAS → конвейер** — единственный способ уложиться в SLA «решение в тот же день» при ограничении batch АБС ↔ конвейер 1 раз/сут (+R3, P4).
3. **Ночной batch предодобрений** — скоринг не перегружается днём (P2, NFR-2).
4. **Повторный скоринг при подаче заявки** — обязателен, если данные клиента устарели (F14).
5. **Java для CAS** — экспертиза в банке (кол-центр, АБС, Camunda); аналогия с DAS упрощает сопровождение.
6. **СМС-модуль в ИБ своими силами** — избегаем зависимости от подрядчика ядра (+R5).

### **Альтернативы**

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| Заявки через АБС → batch → Конвейер | Не укладывается в SLA «решение в тот же день» (P4) |
| Ускорение batch АБС ↔ Конвейер | Технически невозможно (+R3) |
| Предрасчёт скоринга днём для всех клиентов | Перегрузка скоринга в рабочие часы (P2) |
| Прямой вызов конвейера из ИБ/сайта | Нет единой точки для ID заявки и orchestration |
| Прямой online-доступ к Oracle АБС | Перегрузка БД (+R6) |
| Полная замена Camunda | Слишком дорого; конвейер уже масштабируется (R4) |

### **Недостатки, ограничения, риски**

- **Недостаток:** CAS — новая система и точка отказа; нужны мониторинг, резервирование и команда сопровождения.
- **Недостаток:** два пути интеграции с конвейером (online через CAS и batch из АБС для отделений) усложняют поддержку.
- **Ограничение:** batch АБС ↔ Конвейер остаётся для заявок из отделения — чаще 1 раз/сут не ускорить (+R3).
- **Ограничение:** новый клиент с сайта обязан пройти идентификацию в отделении (+R1).
- **Ограничение:** для instant-решения на сайте доступны только данные БКИ (нет внутренней истории нового клиента).
- **Риск:** ночной batch предодобрений может устареть — повторный скоринг при подаче обязателен (F14).
- **Риск:** зависимость от доступности БКИ и скоринга для решения на сайте в момент подачи.
- **Риск:** модуль кредитов и СМС в ИБ — узкое место (команда банка + подрядчик UI).
